这篇的目的不是追某个榜单，而是帮我建立智驾领域的概念地图。智驾系统里很多名词容易混在一起：感知模型、BEV、时序融合、occupancy、online map、prediction、planning、端到端。真正看懂论文前，先要知道它们在系统里解决哪一层问题。

## 总体链路

一个典型智驾栈可以粗略拆成：传感器输入 -> 感知 -> 时序融合/跟踪 -> 地图/场景表示 -> 预测 -> 规划 -> 控制。

- 传感器输入：摄像头、激光雷达、毫米波雷达、IMU、GPS/定位。
- 感知：识别车、人、锥桶、车道线、可行驶区域、交通灯等。
- BEV 表示：把周围环境投到鸟瞰图坐标系里，方便统一处理多相机、多传感器和规划。
- 时序融合：利用历史帧估计速度、遮挡目标、运动趋势。
- 地图/占用：表示道路结构、车道边界、障碍物占据空间。
- 预测：估计其他交通参与者未来会怎么走。
- 规划：输出自车未来轨迹。
- 控制：把轨迹转成方向盘、油门、刹车命令。

## 1. 传统模块化栈

经典思路：感知、预测、规划、控制分别建模，模块之间传递结构化结果，比如 3D box、track、lane、map、trajectory。

优点是可解释、可调试、方便做安全约束；缺点是误差会逐级传递。比如感知漏检一个行人，后面的预测和规划很难凭空修复。

到今天，工业系统仍大量保留模块化思想。所谓端到端也很少真的完全黑盒，通常会保留中间监督、可解释输出和安全后处理。

## 2. LiDAR 3D 检测：PointPillars / CenterPoint

代表论文：

- PointPillars, 2018/2019：https://arxiv.org/abs/1812.05784
- CenterPoint, 2020：https://arxiv.org/abs/2006.11275

为什么经典：早期 3D 感知里，LiDAR 提供直接几何距离，点云检测是高精度基线。PointPillars 把点云柱状化成伪图像，用 2D CNN 高效处理；CenterPoint 把 3D 目标检测转成 BEV 中心点检测。

必须理解：LiDAR 的优势是几何准，深度直接；劣势是成本、稀疏、远距离点少。摄像头路线很多问题，本质是在弥补“没有直接深度”的弱点。

## 3. Camera-only 3D 感知的核心难点

摄像头便宜、信息密度高，但从 2D 图像恢复 3D 是病态问题。一个像素对应一条相机射线，目标到底在近处小物体还是远处大物体，需要模型从语义、几何、多视角和时序里推断。

所以 camera-only 智驾感知绕不开三件事：

- 内外参：像素和真实 3D 空间如何对应。
- 深度估计：把图像特征 lift 到 3D/BEV 时，深度分布是否可靠。
- 多视角融合：环视相机的重叠和盲区如何统一到同一个坐标系。

## 4. Lift-Splat-Shoot

论文：Lift, Splat, Shoot, 2020。链接：https://arxiv.org/abs/2008.05711

核心贡献：提出一条直观的 camera-to-BEV 路线。先对每个图像像素预测深度分布，把 2D 特征 lift 到 3D frustum；再 splat 到 BEV 网格；最后在 BEV 上做任务 head。

干货理解：LSS 是很多 BEV 方法的入门钥匙。它告诉我们：BEV 不是从天上拍的图，而是通过相机几何和深度估计把多视角图像特征“搬运”到地面坐标系。

容易踩坑：深度估计不准会直接污染 BEV。远处目标、小目标、遮挡、夜间场景都会让 lift 过程变差。

## 5. BEVDet

论文：BEVDet, 2021。链接：https://arxiv.org/abs/2112.11790

核心贡献：把多相机 3D 检测明确放到 BEV 范式下做，强调 BEV 空间适合定义 3D 检测目标，也适合接规划。

干货理解：BEVDet 的意义在于工程范式清楚：image backbone 提图像特征，view transformer 转 BEV，BEV encoder 融合，head 输出 3D box。后续很多工作都能放进这个框架比较。

## 6. BEVDepth

论文：BEVDepth, 2022。链接：https://arxiv.org/abs/2206.10092

核心贡献：指出很多 camera BEV 方法的深度是“隐式学出来的假深度”，于是引入显式深度监督、相机参数感知的 depth module 和高效 voxel pooling。

干货理解：BEVDepth 把 camera-only 3D 检测的关键矛盾讲得很清楚：BEV 转换质量很大程度取决于深度。只在最后监督 3D box，模型可能学到可用但不可靠的中间深度。

项目启发：如果项目里发现 3D 检测远近误差大、速度估计不稳、跨相机边界不连续，先查 depth 和 calibration，不要只调 detection head。

## 7. PETR

论文：PETR, 2022。链接：https://arxiv.org/abs/2203.05625

核心贡献：不是显式 lift-splat，而是把 3D 坐标位置编码注入图像特征，让 object query 和 3D position-aware features 交互，做端到端多视角 3D 检测。

干货理解：PETR 代表 query-based 3D detection 路线。它更接近 DETR 思想：用 query 表示要找的对象，用 attention 从多视角特征里取信息。

部署注意：PETR 系列位置编码和 attention 对量化很敏感。2025 年还有 Q-PETR/FQ-PETR 这类工作专门解决 INT8 部署掉点问题，说明好 benchmark 不等于好上车。

## 8. BEVFormer

论文：BEVFormer, 2022。链接：https://arxiv.org/abs/2203.17270

为什么是绝对经典：BEVFormer 把多相机图像、BEV query、spatial cross-attention、temporal self-attention 组合成统一 BEV 表示，是理解后续 BEV 感知和时序模型的核心入口。

核心机制：

- 预定义网格状 BEV queries，每个 query 对应 BEV 平面上的一个位置。
- spatial cross-attention：BEV query 通过相机参数找到图像中的相关区域，从多相机特征聚合空间信息。
- temporal self-attention：把历史 BEV 特征递归融合进当前 BEV，提高速度估计和遮挡鲁棒性。

干货理解：BEVFormer 的关键不是“用了 Transformer”，而是把空间对齐和时间融合都放在 BEV query 空间里做。这样多相机、历史帧和多任务 head 可以共享统一表示。

你需要特别记住：BEVFormer 解决的是 camera-only 多视角 3D 感知的统一表征问题。它不是规划模型，但它生成的 BEV 表示天然适合接检测、地图、占用和规划。

## 9. 时序模型为什么重要

单帧感知看不到速度，遮挡也难处理。智驾里目标是否危险，很大程度取决于时间：前车刹车、行人横穿、旁车加塞、盲区目标出现。

常见时序方式：

- image-space temporal：先在图像特征层融合历史帧。
- BEV-space temporal：把历史 BEV 通过自车位姿补偿后融合。
- recurrent memory：维护历史 BEV memory。
- transformer temporal attention：让当前 query attend 到历史表示。

BEVFormer 属于 BEV-space temporal 的典型代表。TempBEV、BEVPredFormer 等后续工作继续细化“图像空间 + BEV 空间”的时序聚合和未来实例预测。

## 10. Online HD Map：HDMapNet / MapTR

代表论文：

- HDMapNet, 2021：https://arxiv.org/abs/2107.06307
- MapTR, 2022：https://arxiv.org/abs/2208.14437

为什么重要：规划不只需要障碍物，还需要道路拓扑：车道线、边界、人行横道、停止线、方向等。传统高精地图成本高且更新慢，因此 online map construction 很重要。

HDMapNet：从 onboard sensors 预测局部语义地图，建立在线 HD map 的任务和评估框架。

MapTR：把地图元素建模为向量化点集，用 query 和 matching 做端到端学习，解决线状地图元素的顺序歧义问题。

干货理解：地图不是普通分割。车道线、边界、中心线更适合 vector representation，因为规划需要几何结构，而不是只有栅格像素。

## 11. Occupancy Network / SurroundOcc

代表论文：SurroundOcc, 2023。链接：https://arxiv.org/abs/2303.09551

为什么重要：3D box 只能表示已知类别的规则目标，现实世界还有异形障碍物、未知物体、路沿、施工区域。Occupancy 直接预测 3D 空间哪些 voxel 被占据，语义 occupancy 还预测类别。

核心机制：从多相机特征构建 3D volume 或 BEV/voxel 表示，再预测 dense occupancy。SurroundOcc 使用 2D-3D attention 和 3D convolution，并用多帧 LiDAR 融合生成 dense occupancy label。

干货理解：occupancy 更接近“能不能开过去”的安全问题，而不只是“这是不是车”。很多新系统把 occupancy 视为连接感知和规划的关键中间表示。

## 12. Motion Prediction

目标：预测其他 agent 的未来轨迹。输入可能包括历史轨迹、地图、交通规则、邻近交互和感知特征。

关键概念：

- multimodal prediction：同一辆车可能直行、变道、减速，未来不是单一答案。
- interaction modeling：车和车、车和人会互相影响。
- map-aware prediction：车道拓扑限制了可行轨迹。
- planning-aware prediction：预测不是为了好看，而是服务自车安全规划。

干货理解：预测和规划不能完全分开。一个预测模型如果只追 ADE/FDE，可能对规划最关心的碰撞风险不敏感。

## 13. UniAD

论文：Planning-oriented Autonomous Driving, 2023。链接：https://arxiv.org/abs/2212.10156

为什么经典：UniAD 把感知、跟踪、地图、运动预测、占用预测、规划整合成 planning-oriented 的统一框架，强调最终目标是规划质量，而不是单个感知指标。

核心思想：前面任务给后面任务提供结构化信息，最后以 planning 为导向联合优化。它仍保留中间任务监督，不是完全黑盒。

干货理解：智驾模型评价不能只看 detection mAP。更重要的是：感知错误是否影响规划，规划是否安全、舒适、符合交通规则。

## 14. VAD

论文：VAD, 2023。链接：https://arxiv.org/abs/2303.12077

核心贡献：用全向量化场景表示做端到端自动驾驶，把 agent motion 和 map element 都表示成向量，减少 dense raster 的计算成本，并引入向量化规划约束。

干货理解：VAD 的重点是“vectorized scene representation”。相比 occupancy/raster，向量表示更接近车道、边界、轨迹这些规划对象，也更高效。

项目启发：如果项目目标是轨迹规划，不能只问“BEV feature 好不好”，还要问中间表示是否方便表达约束、交互和可解释错误。

## 15. TransFuser

论文：TransFuser, 2022。链接：https://arxiv.org/abs/2205.15997

核心贡献：在端到端驾驶中用 Transformer 融合 camera 和 LiDAR 表示，解决复杂交通场景下多传感器信息互补问题。

干货理解：多传感器融合不只是把特征 concat。什么时候在 image view 融合，什么时候在 BEV 融合，如何处理几何对齐，都会影响鲁棒性。

## 16. 端到端智驾到底是什么

端到端可以有很多层级：

- 输入传感器，直接输出控制：最黑盒，难解释，安全验证难。
- 输入传感器，输出轨迹：比直接控制更常见。
- 多任务中间监督 + 最终规划：目前更实用，兼顾性能和可解释。
- 世界模型/生成式规划：学习环境动态，用 rollout 或生成轨迹辅助决策。

不要把端到端理解成“不要中间结果”。智驾安全要求可诊断、可回放、可约束，所以中间表示仍然非常重要。

## 17. 数据集和指标

常见数据集：nuScenes、Waymo Open Dataset、Argoverse、KITTI、CARLA。

常见指标：

- 3D detection：mAP、NDS、ATE、ASE、AOE、AVE、AAE。
- tracking：AMOTA、MOTA、ID switch。
- map：mAP、Chamfer distance、拓扑相关指标。
- occupancy：IoU、mIoU。
- prediction：minADE、minFDE、miss rate。
- planning：collision rate、L2 trajectory error、driving score、comfort、rule violation。

干货理解：不同指标可能互相冲突。检测 mAP 高，不代表规划安全；轨迹 L2 低，不代表不会碰撞；占用 IoU 高，也不代表边界拓扑一定对。

## 18. 一条建议阅读路线

先读 LSS，理解 camera-to-BEV 的几何直觉。

再读 BEVDet/BEVDepth，理解 camera BEV 3D 检测和深度监督。

再读 PETR/BEVFormer，理解 query-based、attention-based 的多视角 3D 感知和时序融合。

再读 HDMapNet/MapTR，理解在线地图为什么要向量化。

再读 SurroundOcc，理解 occupancy 为什么比 3D box 更贴近安全可行驶空间。

最后读 UniAD/VAD/TransFuser，理解从感知到规划的统一建模和端到端趋势。
