我的工作不是单纯“把模型跑起来”，而是让客户模型在 DCU 上跑得好、跑得快。因此读论文的目的不是背贡献点，而是快速判断：客户模型属于哪条技术路线，核心算子是什么，显存和吞吐瓶颈在哪里，优化到底改变了模型的哪一部分。

这份清单的标准很硬：到 2026 年，如果做 NLP/大模型还不了解这些研究，就很难判断客户在做什么，也很难解释为什么某个 kernel、并行策略、量化策略或图优化会影响效果和性能。

## 先建立工程地图

NLP 模型大致经历了四个阶段：稀疏特征/词向量，RNN/seq2seq，Transformer 预训练，LLM/Agent/多模态。对 DCU 优化最重要的是后两段，但前两段解释了很多概念的来源。

从计算视角看，主流 NLP/LLM 工作负载可以分成：

- encoder-only：BERT、RoBERTa、Embedding、reranker。主要是双向 attention，常见于检索、分类、匹配。
- decoder-only：GPT、LLaMA、Qwen、DeepSeek。主要是 causal attention，训练看全序列，推理高度依赖 KV cache。
- encoder-decoder：T5、BART、翻译/摘要模型。既有 encoder attention，又有 decoder cross-attention。
- sparse/MoE：Switch Transformer、Mixtral、DeepSeek-V3。总参数大，但每 token 只激活部分专家。
- RAG/Agent：模型本体之外还有 embedding、retrieval、rerank、tool call、长上下文拼接。

优化时不要只问“这是大模型吗”。要先问：它是 encoder 还是 decoder？训练还是推理？prefill 还是 decode？dense 还是 MoE？长上下文还是短上下文？瓶颈是 GEMM、attention、通信、KV cache、采样，还是数据加载？

## 1. Word2Vec

论文：Efficient Estimation of Word Representations in Vector Space, 2013。链接：https://arxiv.org/abs/1301.3781

为什么还要知道：它不是今天的性能热点，但它是“离散 token 变连续向量”的经典起点。embedding table、相似度检索、语义空间这些概念都从这里一路延伸到现代 embedding model。

核心机制：CBOW 用上下文预测中心词，Skip-gram 用中心词预测上下文。训练目标把同语境词拉近，负采样降低 softmax 成本。

和 DCU 优化的关系：embedding lookup 本身不是大 GEMM，而是访存型操作；推荐/检索/稀疏特征模型里，embedding 访问、batching、cache locality 往往比算力更关键。看到客户模型大量 embedding、gather、scatter，不要按纯 Transformer 的优化直觉处理。

## 2. Seq2Seq + Attention

代表：Sequence to Sequence Learning with Neural Networks, 2014；Neural Machine Translation by Jointly Learning to Align and Translate, 2014。

为什么经典：它定义了 encoder-decoder 范式，并引入 attention 解决长序列信息压缩问题。后来的 Transformer 本质上是把 attention 机制全面并行化和模块化。

核心机制：encoder 把输入序列编码成隐藏状态，decoder 自回归生成输出；attention 让 decoder 每一步动态查看输入不同位置。

和优化的关系：老式 RNN/seq2seq 的时间步依赖强，难并行，DCU/GPU 利用率常差。客户如果还在用 LSTM/GRU，大概率瓶颈不是单个 matmul，而是小 batch、循环依赖和 kernel launch 频繁。

## 3. Transformer

论文：Attention Is All You Need, 2017。链接：https://arxiv.org/abs/1706.03762

为什么是绝对核心：现代 NLP、LLM、多模态、视觉 Transformer、BEVFormer 这类智驾模型都绕不开它。不了解 Transformer，就很难判断现在绝大多数客户模型的算子结构。

核心机制：multi-head self-attention + FFN + residual + layer norm。attention 负责 token 间交互，FFN 负责逐 token 非线性变换。训练时整段并行，推理时 decoder 逐 token 生成。

工程重点：

- 训练主要吃大 GEMM、attention、激活显存和通信。
- 推理分 prefill 和 decode：prefill 像训练一样吃矩阵计算；decode 每步只生成一个 token，容易被 KV cache 访存、batch 调度和小 GEMM 限制。
- attention 复杂度随序列长度二次增长；KV cache 显存随 batch、层数、head、head_dim、context length 线性增长。

优化含义：FlashAttention、GQA、paged KV cache、continuous batching、算子融合、张量并行，本质都在优化 Transformer 的这些瓶颈。

## 4. BERT

论文：BERT, 2018。链接：https://arxiv.org/abs/1810.04805

为什么必须懂：BERT 是 encoder-only 预训练模型的标准范式。很多企业 NLP 任务不是聊天，而是分类、NER、匹配、排序、embedding、rerank，这些仍大量使用 BERT/RoBERTa/DeBERTa 类模型。

核心机制：双向 Transformer encoder，MLM 预训练，再接任务 head fine-tune。它不是自回归生成模型，不需要 KV cache，也没有逐 token decode。

和 DCU 优化的关系：BERT 类模型推理通常是固定长度或 padding 后 batch，吞吐取决于 batch size、sequence length、attention/FFN GEMM、padding 浪费和动态 shape。优化重点是 padding 策略、batch 合并、layernorm/gelu 融合、matmul 性能和 INT8/BF16 精度。

不要混淆：BERT 的 attention 是 bidirectional，不能直接拿 causal LM 推理优化策略套上去。它更像高吞吐 encoder 服务，而不是 token streaming 服务。

## 5. GPT 与 GPT-3

论文：Language Models are Few-Shot Learners, 2020。链接：https://arxiv.org/abs/2005.14165

为什么必须懂：GPT-3 确立了 decoder-only autoregressive LM + prompt/in-context learning 的主流范式。今天大多数通用大模型服务都沿着这条线。

核心机制：给定前文预测下一个 token。few-shot 示例被放进上下文，不改参数，靠模型在上下文中归纳任务。

工程重点：

- 训练：长序列 causal attention，大规模数据并行/张量并行/流水并行。
- 推理：prefill + decode 两阶段差异巨大。
- decode：每步都读 KV cache，低 batch 时算力吃不满，高 batch 时显存和调度压力大。

优化含义：如果客户说“LLM 推理慢”，要先拆 TTFT 和 TPOT。TTFT 多半看 prefill；TPOT 多半看 decode、KV cache、batching 和采样。

## 6. T5

论文：Exploring the Limits of Transfer Learning with a Unified Text-to-Text Transformer, 2019。链接：https://arxiv.org/abs/1910.10683

为什么必须懂：T5 把所有 NLP 任务统一成 text-to-text，代表 encoder-decoder 预训练路线。翻译、摘要、纠错、信息抽取等客户模型仍可能基于 T5/BART。

核心机制：encoder 读完整输入，decoder 自回归生成输出，并通过 cross-attention 读取 encoder states。

工程重点：encoder-decoder 推理比 decoder-only 多一类 cross-attention KV。输入很长、输出较短时，encoder 侧成本明显；输出很长时，decoder decode 仍是瓶颈。

优化含义：不要把 T5 服务简单按 GPT 服务估算。encoder output 缓存、cross-attention、beam search、batch 内输入/输出长度差异都会影响吞吐。

## 7. Scaling Laws

论文：Scaling Laws for Neural Language Models, 2020。链接：https://arxiv.org/abs/2001.08361

为什么必须懂：客户训练大模型时，参数量、数据量、FLOPs 不是随便拍脑袋。Scaling laws 说明 loss 与模型规模、数据规模、计算量存在幂律关系。

核心机制：用小规模实验预测大规模训练表现，指导 compute budget 分配。

工程含义：你在 DCU 上做训练适配时，客户关心的不只是单步速度，还包括同样预算能训到什么 loss。吞吐、稳定性、可扩展性和断点恢复都会影响最终训练经济性。

优化含义：优化训练吞吐时不能破坏数值稳定。混合精度、通信重叠、梯度累积、ZeRO/FSDP、checkpointing 都要放到“有效训练 FLOPs”里看。

## 8. Chinchilla

论文：Training Compute-Optimal Large Language Models, 2022。链接：https://arxiv.org/abs/2203.15556

为什么必须懂：Chinchilla 改变了“大就是好”的粗糙理解。很多早期模型参数太大、训练 token 太少，本质欠训练。

核心机制：给定训练算力，参数量和训练 token 都要增长，不能只堆参数。

工程含义：客户训练 7B/14B/70B 时，数据 token 数、训练轮数、global batch、学习率 schedule 都会影响是否 compute-optimal。DCU 适配要保证长时间训练稳定，不只是跑通一个 step。

优化含义：数据 pipeline、checkpoint、容错、吞吐稳定性和多机通信同样重要，因为预训练是长期任务。

## 9. LLaMA

论文：LLaMA, 2023。链接：https://arxiv.org/abs/2302.13971

为什么必须懂：LLaMA 让开放权重大模型生态成型。客户私有化部署、LoRA 微调、量化推理、国产硬件适配，经常围绕 LLaMA/Qwen/DeepSeek 这类结构展开。

核心机制：decoder-only Transformer，使用 RoPE、SwiGLU/RMSNorm 等现代组件，充分训练较小参数模型。

工程重点：LLaMA 类模型的性能画像通常是 matmul + attention + RMSNorm + RoPE + KV cache + sampling。优化时要能看懂每层结构，知道哪个 kernel 对 TPOT/TTFT 有影响。

## 10. RoPE

论文：RoFormer, 2021。链接：https://arxiv.org/abs/2104.09864

为什么必须懂：RoPE 是现代 decoder-only LLM 常见位置编码。长上下文扩展、位置插值、NTK scaling、YaRN 等都围绕它。

核心机制：用旋转矩阵把位置信息编码进 query/key，使 attention score 含相对位置信息。

工程含义：RoPE 通常是小算子，但在推理中每层每步都要处理。实现不佳会造成额外 kernel、访存和 dtype 转换开销。长上下文适配时，还要关心位置外推是否改变模型质量。

## 11. FlashAttention

论文：FlashAttention, 2022。链接：https://arxiv.org/abs/2205.14135

为什么必须懂：这是大模型训练和长上下文推理最核心的 attention kernel 思想之一。它不是近似 attention，而是 IO-aware exact attention。

核心机制：通过 tiling 避免显式 materialize 巨大的 attention matrix，减少 HBM 读写，在 SRAM 中分块计算 softmax attention。

工程含义：很多模型理论 FLOPs 不是瓶颈，HBM 访问才是瓶颈。FlashAttention 说明优化要看数据搬运、layout、mask、dropout、causal、varlen，而不是只看公式。

DCU 视角：适配 FlashAttention 类 kernel 时，要重点看 head_dim、seq_len、dtype、causal mask、变长 batch、对齐要求和 fallback 路径。很多性能坑来自“不小心走了通用慢路径”。

## 12. GQA/MQA

论文：GQA, 2023。链接：https://arxiv.org/abs/2305.13245

为什么必须懂：现代 LLM 推理的 KV cache 成本太高，GQA/MQA 是降低 KV head 数、提高 decode 吞吐的主流结构。

核心机制：多个 query heads 共享较少的 key/value heads。MHA 质量好但 KV 大；MQA 最省 KV 但可能掉质量；GQA 折中。

工程含义：同样参数量的模型，GQA 配置不同，服务容量可能完全不同。部署前必须看 `num_attention_heads`、`num_key_value_heads`、`head_dim`。

优化含义：KV cache 分配、paged attention、prefix cache、batch scheduler 都依赖 GQA/MQA 结构。

## 13. MoE / Switch Transformer / DeepSeekMoE

代表：Switch Transformer, 2021；DeepSeek-V3, 2024。链接：https://arxiv.org/abs/2101.03961 和 https://arxiv.org/abs/2412.19437

为什么必须懂：MoE 是 2026 年大模型高性能路线的核心之一。总参数可以很大，但每 token 只激活部分专家。

核心机制：router 为 token 选择专家，专家通常是 FFN。计算量按 activated parameters 走，显存和通信按 total/expert 分布走。

工程重点：

- expert parallel 会带来 all-to-all 通信。
- token dispatch/combine 可能成为瓶颈。
- 负载不均会让某些专家/卡拖慢整体。
- 小 batch decode 下 MoE 可能因为专家分散导致利用率差。

优化含义：MoE 不是“更省的 dense model”。要看路由、专家并行、通信拓扑、batching、capacity factor、负载均衡和 kernel fusion。

## 14. InstructGPT / RLHF

论文：Training language models to follow instructions with human feedback, 2022。链接：https://arxiv.org/abs/2203.02155

为什么必须懂：base model 只会续写，chat/instruct model 才会按用户意图回答。客户模型如果经过 SFT/RLHF，行为和 base model 完全不同。

核心机制：SFT 学示范，reward model 学偏好，PPO 优化策略但限制偏离 base。

工程含义：后训练不一定是最大 FLOPs，但数据格式、loss、序列长度、packing、KL、reward model 都影响训练稳定。做 DCU 训练适配时，要分清 pretrain、SFT、reward model、RLHF 是不同 workload。

## 15. DPO

论文：Direct Preference Optimization, 2023。链接：https://arxiv.org/abs/2305.18290

为什么必须懂：DPO 是大量开源/企业偏好优化的常用方案，比 PPO 式 RLHF 简单得多。

核心机制：用 chosen/rejected pair 直接优化策略相对 reference model 的偏好概率。

工程含义：DPO 通常是双样本序列训练，显存和吞吐受 pair 长度、reference model 前向、loss 实现影响。优化时要注意是否重复计算 reference logits，能否缓存，padding 是否浪费。

## 16. LoRA / QLoRA

论文：LoRA, 2021；QLoRA, 2023。链接：https://arxiv.org/abs/2106.09685 和 https://arxiv.org/abs/2305.14314

为什么必须懂：客户经常不是从零训练，而是在基座模型上做领域适配。LoRA/QLoRA 是最常见的低成本微调方式。

核心机制：LoRA 冻结 base，在部分线性层加低秩矩阵；QLoRA 冻结 4-bit 量化 base，只训练 LoRA adapter。

工程含义：LoRA 训练会改变 matmul 图结构，多出小矩阵乘和 merge/unmerge 逻辑。QLoRA 还涉及量化权重解包、dtype 转换和优化器显存。推理时如果 adapter 未合并，可能增加额外 kernel 和延迟。

优化含义：要看 target modules、rank、是否 merge、量化格式、是否使用 fused LoRA kernel。不要只看模型参数量。

## 17. RAG

论文：Retrieval-Augmented Generation, 2020。链接：https://arxiv.org/abs/2005.11401

为什么必须懂：企业客户大量场景不是“让模型凭记忆回答”，而是检索私有知识后生成。RAG 系统包含 embedding、向量检索、reranker、LLM 生成。

核心机制：外部文档作为非参数记忆，检索相关片段放进上下文，生成模型基于上下文回答。

工程含义：RAG 不是单个 LLM 推理。性能链路包括 embedding batch、向量库延迟、rerank 模型、prompt 拼接、长上下文 prefill。优化 LLM decode 可能不是整体瓶颈。

DCU 视角：embedding/reranker 往往是 BERT-like encoder workload，LLM 是 decoder workload，两者优化策略不同。

## 18. CoT / Self-Consistency / Reasoning Models

代表：Chain-of-Thought, 2022；Self-Consistency, 2022；DeepSeek-R1, 2025。链接：https://arxiv.org/abs/2201.11903、https://arxiv.org/abs/2203.11171、https://arxiv.org/abs/2501.12948

为什么必须懂：2025 以后 reasoning model 把 test-time compute 推到核心位置。模型可能不是一次短回答，而是长思考、多采样、验证、再输出。

核心机制：CoT 用中间推理 token 提升复杂任务；self-consistency 采样多条路径投票；R1 说明可验证任务上的 RL 能诱导推理行为。

工程含义：推理成本不再只由输出长度决定，还由思考 token、采样条数、验证器、best-of-N、工具调用决定。客户如果做数学/代码/智能体，TPOT 和总 token 成本会非常高。

## 19. Toolformer / ReAct / Agent

代表：Toolformer, 2023；ReAct, 2022。链接：https://arxiv.org/abs/2302.04761 和 https://arxiv.org/abs/2210.03629

为什么必须懂：很多客户应用已经不是单轮问答，而是模型调用搜索、数据库、代码执行、业务 API。

核心机制：ReAct 让模型交替推理和行动；Toolformer 学习何时调用工具以及如何使用返回结果。

工程含义：Agent 性能瓶颈常在系统层：多轮 LLM 调用、工具 I/O、上下文膨胀、失败重试、权限校验。硬件优化只能解决其中一部分。

## 20. Mamba / 非 Transformer 路线

论文：Mamba, 2023。链接：https://arxiv.org/abs/2312.00752

为什么要知道：Transformer 仍是主流，但长序列和低延迟需求推动 SSM、linear attention、hybrid model。客户模型不一定永远是标准 attention。

核心机制：selective state space model 根据输入动态选择保留/遗忘信息，序列长度上线性扩展。

工程含义：非 Transformer 模型的瓶颈可能是 scan、conv、state update，而不是 attention/KV cache。适配时不能只准备 Transformer kernel。

## 面向 DCU 优化的判断清单

拿到客户模型，先问这些问题：

1. 模型类型：encoder-only、decoder-only、encoder-decoder、MoE、RAG、Agent、多模态？
2. 运行阶段：pretrain、SFT、DPO/RLHF、embedding、rerank、prefill、decode？
3. 核心形状：batch、seq_len、hidden、layers、heads、kv_heads、vocab、expert 数？
4. 精度：FP32、FP16、BF16、INT8、INT4、混合精度是否影响收敛或输出？
5. 瓶颈：GEMM、attention、KV cache、all-reduce、all-to-all、embedding lookup、sampling、数据加载？
6. 图结构：是否有 LoRA、quant、MoE router、custom op、dynamic shape、beam search、变长输入？
7. 指标：训练 tokens/s，推理 TTFT/TPOT/QPS，端到端延迟，显存峰值，质量指标是否保持？

真正合格的优化不是“某个 kernel 快了”，而是能解释：这个模型为什么慢，慢在什么阶段，优化改变了哪条计算/访存/通信路径，以及为什么不改变模型语义。
