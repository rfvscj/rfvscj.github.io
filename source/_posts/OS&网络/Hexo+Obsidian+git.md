通过这种方式实现笔记同步和博客写作，是这套博客工作流的起点。

## 工作流概览

- Obsidian 负责日常写作和知识管理
- Hexo 负责静态站点生成
- Git 负责版本管理和部署

这套组合的好处，不是“技术上很炫”，而是它把写作、整理、发布拆成了三个清晰环节：

- Obsidian 负责低摩擦记录
- Hexo 负责稳定生成网页
- Git 负责把内容同步到远端并保留历史

## 为什么这套组合适合个人博客

### 1. 写作体验比直接在博客后台舒服

很多静态博客的痛点是“发布方便，但写作不顺手”。Obsidian 刚好反过来，它非常适合快速记笔记、建立知识网络、用 Markdown 持续写长文。

### 2. Hexo 把“笔记”变成“站点”

Obsidian 解决的是写作，不解决公开展示。Hexo 的作用是把 Markdown、分类、标签、页面配置统一生成成一个静态网站。

### 3. Git 解决同步和回滚

Git 在这里不是“为了显得专业”，而是确实好用：

- 可以记录每次修改
- 可以在多台电脑之间同步
- 写坏了能回滚
- 部署到 GitHub Pages 也很顺手

## 目录怎么组织

比较顺手的一种方式是：

- Obsidian 仓库里保留日常笔记和草稿
- 真正准备公开的文章，再整理进 `source/_posts/`
- 图片统一放在博客仓库自己的 `source/img/` 或文章约定目录里

关键原则是：不要把 Obsidian 里的所有碎片化笔记自动发布。Obsidian 是工作区，Hexo 是发布区。

## 图片和附件怎么放

如果文章最终会被公开发布，最好尽早规范图片路径。对个人博客来说，统一放在 `source/img/` 通常更简单，因为路径直观、迁移方便，也不容易因为粘贴图片时的相对路径而失效。

## 本地预览和部署

在博客仓库里最常见的命令就是：

```bash
npm install
npm run clean
npm run server
npm run build
npm run deploy
```

其中：

- `server` 用于本地预览
- `build` 用于检查静态生成是否正常
- `deploy` 用于发布到远端

## 适合什么样的写作流程

1. 在 Obsidian 中随手记素材
2. 把值得公开的内容整理成一篇完整文章
3. 移入 Hexo 的 `source/_posts/`
4. 补齐 front matter、分类、标题和图片
5. 本地预览
6. 提交并部署

## 这套方案的局限

它也不是没有代价：

- 你要理解 Git 和静态站点的基本流程
- 图片路径、站内链接、文章 front matter 需要有一点纪律
- 如果仓库维护得很随意，时间长了也会变乱

## 总结

`Obsidian + Hexo + Git` 的价值，不是它们分别多强，而是三者分工明确：Obsidian 管输入，Hexo 管展示，Git 管历史和发布。它很适合长期写技术博客，前提是你愿意多做一步整理，而不是把所有笔记直接往外倒。
